## BOUNDED CONTEXT MAP

### BC1: Restaurant & Menu Context
- Domain: Restaurant catalog management, menu curation, promotions
- Entities: Restaurant, MenuItem, Promotion
- Aggregate Root: Restaurant
- Domain Events:
  - RestaurantCreated, RestaurantUpdated, RestaurantDeactivated
  - MenuItemAdded, MenuItemUpdated, MenuItemAvailabilityChanged
  - PromotionCreated, PromotionExpired
- Business Rules:
  - Restaurant must have at least one active menu item to be listed
  - MenuItem price must be positive
  - Promotions have date-bounded validity
  - CommissionRate is set per restaurant (0-100%)
- Data Store: Dedicated SQL database (restaurant-db)
- Scaling Profile: Read-heavy (menu browsing), cache-friendly

### BC2: Order Context
- Domain: Order lifecycle management, order items, pricing
- Entities: Order, OrderItem
- Aggregate Root: Order
- Domain Events:
  - OrderPlaced, OrderConfirmed, OrderPaid, OrderAssigned
  - OrderPickedUp, OrderDelivered, OrderCancelled, OrderTimedOut
- Business Rules:
  - Order requires valid customer and active restaurant
  - All menu items must be available at time of order
  - Order total calculated with commission and promotions
  - Pending orders timeout after 30 minutes
  - Status transitions: Pending → Paid → Assigned → PickedUp → Delivered
  - Status transitions: Pending → Cancelled (timeout or user action)
- Data Store: Dedicated SQL database (order-db)
- Scaling Profile: Write-heavy during peak hours, event-sourcing candidate

### BC3: Payment Context
- Domain: Payment processing, refunds, financial records
- Entities: Payment
- Aggregate Root: Payment
- Domain Events:
  - PaymentInitiated, PaymentSucceeded, PaymentFailed, PaymentRefunded
- Business Rules:
  - Payment requires valid order with calculated total
  - Stripe is the payment processor (abstracted behind interface)
  - Payment amount must match order total
  - Successful payment triggers order status update
  - Payment records are immutable (append-only audit trail)
- Data Store: Dedicated SQL database (payment-db)
- Scaling Profile: Bursty writes, strict consistency, PCI scope
- External Dependencies: Stripe API

### BC4: Delivery Context
- Domain: Driver management, delivery tracking, route optimization
- Entities: Delivery, Driver
- Aggregate Root: Delivery
- Domain Events:
  - DriverRegistered, DriverAvailable, DriverBusy
  - DeliveryCreated, DeliveryAssigned, DeliveryPickedUp
  - DeliveryLocationUpdated, DeliveryCompleted
- Business Rules:
  - Driver assignment picks highest-rated available driver
  - Assigned driver becomes unavailable
  - Delivery tracks real-time location updates
  - DeliveryHub provides WebSocket streaming
- Data Store: Dedicated database (delivery-db), potentially Redis for location
- Scaling Profile: Real-time updates, location-heavy, WebSocket connections
- External Dependencies: SignalR / Azure SignalR Service

### BC5: Customer/User Context
- Domain: Customer profiles, authentication, loyalty
- Entities: Customer
- Aggregate Root: Customer
- Domain Events:
  - CustomerRegistered, CustomerUpdated, LoyaltyPointsEarned
- Business Rules:
  - Email must be unique and valid
  - LoyaltyPoints accumulate per completed order
  - DefaultAddress used when no delivery address specified
- Data Store: Dedicated SQL database (customer-db)
- Scaling Profile: Read-heavy (profile lookups), auth caching

### BC6: Notification Context
- Domain: Multi-channel notifications (email, push, SMS)
- Entities: None (stateless, event-driven)
- Domain Events Consumed:
  - OrderPlaced → Send order confirmation email
  - OrderStatusChanged → Send status update email
  - DeliveryAssigned → Send driver assignment notification
- Business Rules:
  - Notifications are fire-and-forget (async)
  - Failed notifications should retry with backoff
  - Template-based message generation
- Data Store: None (or simple outbox for retry)
- Scaling Profile: Event-driven, horizontally scalable
- External Dependencies: SendGrid (email), future: Twilio (SMS)

```
[Order] ←── customer-supplier ──→ [Delivery]
    Order publishes OrderPaid, Delivery assigns driver and tracks

[Customer] ←── conformist ──→ [Order]
    Order validates customer exists via API

[Order] ──── publishes events ──→ [Notification]
    Notification subscribes to order lifecycle events

[Delivery] ──── publishes events ──→ [Notification]
    Notification subscribes to delivery events
```

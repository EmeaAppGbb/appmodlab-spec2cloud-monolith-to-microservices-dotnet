## STRANGLER FIG MIGRATION PLAN

The Strangler Fig pattern incrementally replaces monolith functionality with
microservices while keeping the monolith operational throughout migration.

## PHASE 0: PREPARATION (Week 1-2)

### 0.1 Set Up Infrastructure
- Provision Azure Container Apps environment
- Create Azure Service Bus namespace with topics
- Set up Azure SQL databases (one per future service)
- Configure Azure API Management as facade
- Set up Azure Container Registry
- Configure CI/CD pipelines (GitHub Actions)

### 0.2 API Gateway Facade
- Deploy API Management in front of the monolith
- Route ALL traffic through the gateway
- Establish baseline metrics (latency, error rates, throughput)
- Client applications update to use gateway URL

### 0.3 Upgrade .NET Version
- Migrate monolith from netcoreapp3.1 → .NET 8
- Update all NuGet packages to latest compatible versions
- Verify all tests pass after upgrade
- This is prerequisite — 3.1 is end-of-life

## PHASE 1: NOTIFICATION SERVICE (Week 3-4) — Lowest Risk

### Why First
- Stateless, no database migration needed
- Already loosely coupled (only called by OrderService)
- Fire-and-forget pattern — easy to verify

### Migration Steps
1. Create UrbanBites.Notification microservice (.NET 8)
2. Implement SendGrid email sending (from NotificationService.cs)
3. Add Azure Service Bus consumer for order events
4. Deploy to Azure Container Apps
5. Modify monolith OrderService to publish events instead of calling NotificationService directly
6. Verify emails still arrive for order confirmation and status updates
7. Remove NotificationService from monolith DI registration

### Rollback: Re-enable direct NotificationService call in monolith

## PHASE 2: RESTAURANT/MENU SERVICE (Week 5-7) — Read-Heavy, Clear Boundary

### Why Second
- Mostly read operations (menu browsing)
- Clear data ownership (Restaurant, MenuItem, Promotion, Review)
- No transactional coupling with order flow

### Migration Steps
1. Create UrbanBites.Restaurant microservice (.NET 8)
2. Migrate Restaurant, MenuItem, Promotion, Review entities
3. Create RestaurantDbContext with dedicated database
4. Implement all RestaurantController + ReviewController endpoints
5. Data migration: copy restaurant data to new database
6. Set up CDC (Change Data Capture) for transition period
7. Update API Gateway to route /api/restaurants/* to new service
8. Update monolith OrderService to call Restaurant API for validation
9. Verify menu browsing, restaurant creation, review creation
10. Remove RestaurantController, ReviewController, ReviewService from monolith

### Rollback: Switch API Gateway route back to monolith

## PHASE 3: CUSTOMER SERVICE (Week 8-9) — Simple CRUD

### Why Third
- Simple entity with minimal business logic
- Only consumed by Order for validation
- Prepares for Order service extraction

### Migration Steps
1. Create UrbanBites.Customer microservice (.NET 8)
2. Migrate Customer entity to CustomerDbContext
3. Implement CRUD endpoints + loyalty points
4. Data migration: copy customer data
5. Update API Gateway routing
6. Update monolith OrderService to call Customer API for validation
7. Remove CustomerController from monolith

### Rollback: Switch API Gateway route back

## PHASE 4: PAYMENT SERVICE (Week 10-12) — PCI Compliance Driver

### Why Fourth
- Isolates PCI scope to a single service
- Event-driven communication with Order
- Stripe integration moves cleanly

### Migration Steps
1. Create UrbanBites.Payment microservice (.NET 8)
2. Migrate Payment entity to PaymentDbContext
3. Move Stripe integration (use Azure Key Vault for secrets)
4. Implement payment processing endpoint
5. Add Service Bus: subscribe to OrderPlaced, publish PaymentSucceeded/Failed
6. Data migration: copy payment records
7. Update API Gateway routing
8. Monolith Order publishes OrderPlaced event instead of calling PaymentService
9. Verify payment flow end-to-end
10. Remove PaymentService, PaymentController from monolith

### Rollback: Re-enable direct PaymentService call

## PHASE 5: DELIVERY SERVICE (Week 13-16) — Real-Time, Complex State

### Why Fifth
- Real-time tracking requires dedicated infrastructure
- SignalR hub migration to Azure SignalR Service
- Background job migration (DeliveryAssignmentJob)

### Migration Steps
1. Create UrbanBites.Delivery microservice (.NET 8)
2. Migrate Delivery, Driver entities to DeliveryDbContext
3. Migrate DeliveryHub to Azure SignalR Service
4. Replace Hangfire DeliveryAssignmentJob with Service Bus consumer
5. Subscribe to OrderPaid event → auto-assign driver
6. Publish DriverAssigned, DeliveryLocationUpdated, DeliveryCompleted events
7. Data migration: copy delivery and driver data
8. Update API Gateway routing
9. Verify real-time tracking, driver assignment
10. Remove DeliveryService, DeliveryController, DeliveryHub, DeliveryAssignmentJob from monolith

### Rollback: Re-enable monolith delivery components

## PHASE 6: ORDER SERVICE (Week 17-20) — Last, Most Coupled

### Why Last
- Orchestrator of the order flow
- Most cross-domain dependencies
- By now all dependencies are microservices with APIs/events

### Migration Steps
1. Create UrbanBites.Order microservice (.NET 8)
2. Migrate Order, OrderItem entities to OrderDbContext
3. Implement order placement with HTTP calls to Restaurant + Customer APIs
4. Implement event publishing (OrderPlaced, OrderPaid, etc.)
5. Subscribe to PaymentSucceeded/Failed events
6. Replace Hangfire OrderTimeoutJob with background hosted service
7. Merge PricingService logic into Order service (calls Restaurant API for promotions)
8. Data migration: copy order data
9. Update API Gateway routing
10. Full end-to-end verification
11. Decommission monolith

### Rollback: Re-enable monolith Order components

## PHASE 7: DECOMMISSION MONOLITH (Week 21-22)

1. Verify all traffic routed through microservices (zero monolith traffic)
2. Run parallel for 1 week with monitoring
3. Remove monolith from API Gateway
4. Archive monolith codebase
5. Clean up shared database (no more cross-service queries)
6. Remove Hangfire database

## RISK MITIGATION

### Data Consistency
- Use Saga pattern for distributed transactions (Order → Payment → Delivery)
- Compensating transactions for failures
- Outbox pattern for reliable event publishing

### Feature Flags
- Use Azure App Configuration feature flags
- Toggle between monolith and microservice per endpoint
- Gradual rollout (1% → 10% → 50% → 100%)

### Monitoring
- Distributed tracing with Application Insights
- Custom dashboards per service
- Alert on error rate increase during migration phases
- Automated rollback triggers

### Testing Strategy
- Contract tests between services (Pact)
- Integration tests with TestContainers
- End-to-end smoke tests after each phase
- Load testing to verify scaling behavior

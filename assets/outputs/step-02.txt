## DECOMPOSITION ANALYSIS

The UrbanBites monolith exhibits classic indicators for microservice decomposition:
- Single deployment unit with 6 distinct business domains
- God DbContext with 10 entities spanning unrelated bounded contexts
- Cross-domain coupling making independent scaling impossible
- Mixed SLA requirements (real-time delivery vs batch reviews)

## CANDIDATE MICROSERVICES IDENTIFIED

### 1. Restaurant/Menu Service (High Priority)
- Entities: Restaurant, MenuItem, Promotion
- Current: RestaurantController directly queries DbContext
- Rationale: Menu browsing is read-heavy, needs independent scaling
- Change frequency: Menu updates are frequent, independent of orders
- Data ownership: Restaurant catalog is a natural bounded context

### 2. Order Service (High Priority)
- Entities: Order, OrderItem
- Current: OrderService with cross-domain dependencies
- Rationale: Core business flow, needs transactional integrity
- Change frequency: Business rules change often (pricing, validation)
- Data ownership: Order lifecycle is self-contained after placement

### 3. Payment Service (High Priority)
- Entities: Payment
- Current: PaymentService with Stripe integration
- Rationale: PCI compliance scope reduction, independent scaling
- Change frequency: Payment provider changes are isolated
- Data ownership: Payment records are append-only, audit trail

### 4. Delivery Service (Medium Priority)
- Entities: Delivery, Driver
- Current: DeliveryService + DeliveryHub + DeliveryAssignmentJob
- Rationale: Real-time tracking has different scaling needs
- Change frequency: Driver assignment algorithms evolve independently
- Data ownership: Delivery state is transient, location-heavy

### 5. Customer/User Service (Medium Priority)
- Entities: Customer
- Current: CustomerController with basic CRUD
- Rationale: User management is a cross-cutting concern
- Change frequency: Auth/profile changes are independent
- Data ownership: Customer profiles, loyalty points

### 6. Notification Service (Low Priority — Extract First)
- Entities: None (stateless)
- Current: NotificationService with SendGrid
- Rationale: Already loosely coupled, easiest to extract
- Change frequency: Notification channels change independently
- Data ownership: No persistent state, event-driven

## COUPLING ANALYSIS

### Current Dependency Graph
```
OrderController → OrderService → PaymentService
                               → NotificationService
                               → PricingService → DbContext (Restaurants, Promotions)
                               → DbContext (Customers, Restaurants, MenuItems, Orders)
DeliveryAssignmentJob → DeliveryService → DbContext (Drivers, Deliveries)
                      → DbContext (Orders)
OrderTimeoutJob → DbContext (Orders)
ReviewController → DbContext (Reviews)
RestaurantController → DbContext (Restaurants, MenuItems)
```

### Problematic Couplings
1. OrderService reads Customer, Restaurant, MenuItem — should use APIs
2. PricingService reads Restaurant.CommissionRate + Promotions — cross-context
3. DeliveryAssignmentJob reads Orders + writes Deliveries — cross-service
4. OrderTimeoutJob directly mutates Order status — should emit events
